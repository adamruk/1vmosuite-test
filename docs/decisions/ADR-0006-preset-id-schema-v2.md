# ADR-0006: Preset identity model — prefix-namespaced IDs + schema_version=2

**Status:** Accepted

**Date:** 2026-04-27

**Decision makers:** Adam (primary), via PARALLEL discovery 2026-04-27

## Context

Phase 2c-c-3 (commit `86edaa4`) shipped EncoderDialog with the D3=a
decision: any preset edit creates a user entry; no UI distinction
between built-in and user; identity remained position-based via
`_builtin_preset_count` boundary marker.

PARALLEL discovery for 2c-c-4 surfaced a silent-data-loss bug
introduced by D3=a: `edit_encoder` mutates `self.encoder_options[idx]`
in-place. If the index lands within the built-in range, the mutation
appears to persist in-memory but `save_encoder_changes` slices
`[builtin_count:]` so the change is never written to disk. User
closes the app, reopens, the edit is gone — no warning, no log
message.

Same bug pattern in `delete_encoder`: pop from built-in range
appears to remove the preset until next launch.

The bug exists because identity is position-based instead of
intrinsic. The plan (`docs/PHASE_2C_PLAN.md` backlog 2c-c-4)
already called for prefix-namespaced IDs; this ADR documents the
identity model decisions.

Additionally: 13 collisions exist across 111 entries in
`assets/Encoder.json` (10 entries named "Layer Overlay" in
"🕹️ 1vmo Ultimate" group, 3 entries named "Add UI Divider Line"
with empty group). The composite (group, name) tuple is not unique;
identity-by-composite is broken for these 13 entries today.

ADR-0002 § Out-of-scope had named "ID format" as a versioned
decision deferred to a later ADR. This ADR is that decision.

## Decision

### Schema version bump: 1 → 2

`EncoderLibrary.schema_version` is bumped from `Literal[1]` to
`Literal[2]`. Built-in JSON files (`assets/Encoder.json`) are
regenerated with v2 envelope. User JSON files
(`encoder.user.json`) are lazy-migrated on first load: load v1
successfully, derive IDs, write v2 on next save. v1 user files
with manually-set v1 fields are preserved through the migration.

### ID field on EncoderPreset

New required field: `id: str`. Format regex enforces asymmetry
between builtin and user namespaces:

    ^(builtin:([a-z0-9-]+/)?[a-z0-9-]+(-\d+)?|user:[a-z0-9-]+(-\d+)?)$

- `builtin:foo` ✓ (no group prefix)
- `builtin:group/foo` ✓ (with group prefix)
- `builtin:group/foo-3` ✓ (with collision suffix)
- `user:foo` ✓ (flat namespace)
- `user:foo-2` ✓ (with collision suffix)
- `user:group/foo` ✗ (user namespace is flat per D3 plan-literal)

Pydantic Field validator enforces format. Library-level validator
enforces uniqueness across all presets in a library.

### Slug derivation rule

Five-step transform applied to both group and name:

1. NFKD-normalize unicode; strip combining marks.
2. Lowercase.
3. Replace any run of non-[a-z0-9] characters with single hyphen.
4. Strip leading/trailing hyphens.
5. Collapse repeated hyphens (subsumed by step 3 since runs are
   replaced with a single hyphen).

Worked examples:

- "Scale 99%" → "scale-99"
- "Cycle 10s (4-3-3) 100x Zoom" → "cycle-10s-4-3-3-100x-zoom"
- "9:16 CRF High" → "9-16-crf-high"
- "🕹️ 1vmo Ultimate" → "1vmo-ultimate"

### Prefix syntax

- Built-in: `builtin:<group-slug>/<name-slug>` if group is
  non-empty, `builtin:<name-slug>` if group is empty.
- User: `user:<name-slug>`. User presets have no group prefix
  (user-authored presets opt into a flat namespace per plan
  backlog literal text).

### Disambiguation on collisions

Within a library, when two or more entries produce the same base
slug (same group-slug and same name-slug), append `-N` where N is
the 1-indexed position in the file-order collision cluster. Unique
slugs receive no suffix.

Example:
- 10 "Layer Overlay" entries in "🕹️ 1vmo Ultimate":
  - `builtin:1vmo-ultimate/layer-overlay-1`
  - `builtin:1vmo-ultimate/layer-overlay-2`
  - ... through `builtin:1vmo-ultimate/layer-overlay-10`
- 3 "Add UI Divider Line" entries with empty group:
  - `builtin:add-ui-divider-line-1`
  - `builtin:add-ui-divider-line-2`
  - `builtin:add-ui-divider-line-3`
- Unique entry "Scale 99%" stays `builtin:1vmo-ultimate/scale-99`
  (no suffix).

Suffix is computed at load time. Does not require Encoder.txt edits.

### Lazy migration semantics

- `assets/Encoder.json`: regenerated by
  `tools/generate_encoder_json.py` with v2 envelope + ids on every
  entry. Schema version validation in `load_builtin_json` requires
  v2; loud-fail on v1 (consistent with current behavior).
- `encoder.user.json`: load supports v1 (no id field, derive at
  load time) AND v2 (read id as-is). On first save after a v1
  load, file is rewritten as v2. Migration is invisible to user
  beyond a single info-level log line.
- `config_video_renderer.json` `encoder_options` field: not
  consumed at app startup in current codebase (selected_encoders
  is repopulated from current tree selection at render-time). No
  migration needed; old data is dead-on-disk and overwritten on
  next save.

### EncoderDialog UI changes (eliminates silent-data-loss bug)

- Built-in entries display name in italic font in the tree column.
- Edit and Delete buttons disabled when selection includes any
  built-in entry. Selection-change handler toggles button state.
- Add button always enabled.
- Built-in vs user determined by `id.startswith("builtin:")`.

### Architectural cleanup

`_builtin_preset_count` instance attribute (added in 2c-c-3) is
removed. Built-in vs user is intrinsic via id prefix; position-
based boundary is no longer needed.

### derive_ids_for_presets API

Helper takes a list of `(group, name)` tuples and returns a list
of ids. Operates on tuples (not Preset objects) to avoid the
chicken-and-egg of constructing a Preset (which requires id) just
to derive its id. Callers parse to intermediate tuples first,
derive ids, then construct Presets in one go.

## Consequences

### Positive

- Silent-data-loss bug from 2c-c-3 D3=a is fixed: built-ins are
  visibly read-only, edits prevented at UI layer.
- Identity is model-layer, not position-layer: 13 collision-cluster
  entries are uniquely addressable for the first time.
- User presets can be referenced by stable id from external code
  (e.g., bundled `_internal/config_video_renderer.json` slot
  defaults can name presets by id rather than fragile full_name
  composites).
- Schema bump anchors the decision; future schema changes follow
  ADR-0002's lazy-migration framework.

### Negative

- Schema bump is a breaking change for any external consumer of
  `assets/Encoder.json` v1 (none known; 1vmo Suite is the only
  consumer).
- Slug rule is opinionated; renaming a preset in Encoder.txt
  causes its slug (and id) to change, breaking any external
  reference. Mitigation: rename of shipped presets is governed by
  PRESET_PHILOSOPHY.md rules; rare event.
- 13 collision-cluster entries get suffixed ids that look like
  "layer-overlay-1" through "-10" — readable but not as clean as
  descriptive names. D7=a defers preset renaming to 2c-e Part B.
- Minor: `_builtin_preset_count` removal is a small refactor; any
  code depending on it (only `save_encoder_changes` today) is
  rewritten in this commit.

### Mitigations / future evolution

- If preset renaming becomes desirable (e.g., "Layer Overlay 1"
  → "Layer Overlay (Subtle)"), 2c-e Part B audit handles it.
- If user-authored presets need group prefixes, schema can extend
  to `user:<group-slug>/<name-slug>` in v3 without breaking v2.
